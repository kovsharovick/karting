# Нефункциональные требования

> Этап 2. Как система должна работать: платформа, интеграция, надёжность, ограничения проекта.
> Не путать с функциональными ([functional-requirements.md](functional-requirements.md)).

> **Скоуп — клиентское мобильное приложение и API для него.**

| ID | Категория | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- | :-- |
| NFR-1 | Удобство использования | Интерфейс клиентского приложения — **mobile-first**, рассчитан на использование с телефона непосредственно у трассы. | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Технические / платформенные» |
| NFR-2 | Интеграция | Клиентское приложение взаимодействует с **существующим бэкендом через API**; слоты, конфигурации трасс и маршалы для клиента доступны только для чтения. | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Технические / платформенные», «Границы скоупа» |
| NFR-3 | Надёжность / Целостность данных | Гарантия «**0 двойных броней**» обеспечивается на стороне существующего бэкенда (атомарная проверка свободных картов при бронировании); клиентское приложение полагается на ответы API и корректно обрабатывает отказ, если мест уже нет (R-004). | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Граница серверной интеграции» (R-004) |
| NFR-4 | Интеграция / Данные | Проект учебный/тестовый, легаси-данных нет; **канонической схемой считается контракт API** — бэкенд отдаёт все поля контракта, включая прокатный тариф, конфигурацию трассы, адрес центра, место сбора и свободный прокатный фонд экипировки (R-015). | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Данные существующей инфраструктуры» (R-015) |
| NFR-5 | Безопасность / Аутентификация | Авторизация клиента — номер телефона и одноразовый код в SMS, **без паролей**. | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Технические / платформенные» |
| NFR-6 | Безопасность / Разграничение доступа | Клиент через клиентское приложение видит и управляет **только своими** бронированиями; данные расписания и справочники доступны read-only. | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Пользователи и роли», «Границы скоупа» |
| NFR-7 | Ограничения проекта | Решение должно быть запущено к **старту высокого сезона** — жёсткий срок ~**2 месяца**. | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Проект» |
| NFR-8 | Ограничения проекта | **Бюджет ограничен** → MVP-подход: начинаем с самого нужного, «навороты» — позже. | Высокий (Must) | [domain-description.md](../1-elicitation/domain-description.md) → «Проект», «Границы скоупа» |
| NFR-9 | Архитектура / Границы | Детали серверной реализации (транзакционность, SLA, маппинг сущностей) **вне скоупа** клиентского приложения; существующий бэкенд — black-box источник истины (R-004). | Средний | [domain-description.md](../1-elicitation/domain-description.md) → «Граница серверной интеграции» (R-004) |
