# Use cases / сценарии

> Этап 2. Сценарии использования.
>
> **Скоуп: клиентское приложение и API для него.** Слоты, конфигурации трасс и маршалы поступают
> из существующего бэкенда через API (read-only).

## UC-1: Авторизация клиента (SMS OTP)

- **Актёры:** Клиент
- **Предусловия:** У клиента есть мобильный телефон с возможностью получить SMS.
- **Триггер:** Клиент открывает приложение для записи на заезд.
- **Приоритет:** Must
- **Источник:** [domain-description.md](../1-elicitation/domain-description.md) → «Технические / платформенные», «Границы скоупа»
- **Связанные FR:** FR-1

### Основной поток
1. Клиент указывает номер телефона.
2. Система отправляет одноразовый код в SMS.
3. Клиент вводит код.
4. Система авторизует клиента без пароля.

### Исключения / ошибки
- **E1. Неверный код.** Система сообщает об ошибке; клиент может повторить ввод.
- **E2. Код не доставлен.** Клиент может запросить повторную отправку (детали — на стороне API/инфраструктуры, вне скоупа клиента).

### Постусловия
- Клиент авторизован и может просматривать слоты и управлять своими бронями.

## UC-2: Просмотр и фильтрация слотов

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет найти подходящий заезд.
- **Приоритет:** Must
- **Источник:** [domain-description.md](../1-elicitation/domain-description.md) → «Процесс», «Бизнес-ограничения» (R-027), «Границы скоупа»
- **Связанные FR:** FR-2, FR-3, FR-4, FR-5, FR-13

### Основной поток
1. Клиент открывает список слотов на **ближайшие 7 дней** (R-027).
2. (Опц.) Задаёт фильтры: дата, конфигурация трассы, наличие мест, маршал.
3. Система показывает отфильтрованный список с ключевыми параметрами слотов.
4. Клиент открывает карточку слота и просматривает дату/время, конфигурацию трассы, маршала и его средний рейтинг, всего/свободно мест, предупреждение по возрасту/росту, итоговую стоимость, адрес и место сбора.

### Альтернативные потоки
- **A1. Расширенный период.** Клиент задаёт фильтр дат на период длиннее 7 дней.
- **A2. Сброс фильтров.** Клиент очищает фильтры → возвращается выдача по умолчанию (7 дней).

### Исключения / ошибки
- **E1. Нет слотов в горизонте 7 дней.** Показывается empty state «Пока нет доступных заездов».
- **E2. Фильтр не дал результатов.** Показывается пустой список с возможностью изменить фильтры.

### Постусловия
- Клиент видит актуальный список/карточку слотов; данные расписания не изменяются.

## UC-3: Запись на заезд (себя и гостей)

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; выбран слот со свободными местами; слот не отменён центром.
- **Триггер:** Клиент хочет забронировать место(а) на заезде.
- **Приоритет:** Must
- **Источник:** [domain-description.md](../1-elicitation/domain-description.md) → «Процесс», «Бизнес-ограничения», «Граница серверной интеграции» (R-004, R-008)
- **Связанные FR:** FR-6–FR-12

> **Бизнес-решение.** По гостям собираются только **число мест и выбор экипировки** (своя / прокат).
> Персональные данные гостей не собираются.

### Основной поток
1. Клиент выбирает слот и указывает число мест (себя + гостей).
2. Для каждого места выбирает экипировку: своя или прокатная.
3. Система показывает итоговую стоимость из API (оплата офлайн).
4. Клиент подтверждает запись.
5. Система (через API) проверяет лимит мест `min(свободные карты, потолок конфигурации трассы)` и отдельно — прокатный фонд; создаёт бронь.
6. Система показывает подтверждение; бронь появляется в «Моих бронированиях».

### Альтернативные потоки
- **A1. Вся экипировка своя.** Прокатный фонд не уменьшается; уменьшается только число мест в группе.
- **A2. Часть мест — прокат.** Уменьшается прокатный фонд только на число прокатных комплектов.

### Исключения / ошибки
- **E1. Не хватает мест.** Запрошено больше, чем доступно → запись отклоняется, показывается актуальное число свободных мест.
- **E2. Не хватает прокатной экипировки.** Прокат запрошен сверх свободного фонда → система предлагает уменьшить прокат или выбрать «своя экипировка».
- **E3. Гонка запросов / слот заполнился.** Места заняты другим клиентом → запись отклоняется без овербукинга (R-004).
- **E4. Слот отменён центром.** Повторная запись на отменённый слот запрещена (R-008).

### Постусловия
- При успехе: создана бронь; счётчики мест/проката уменьшены на стороне бэкенда.
- При ошибке: бронь не создана.

## UC-4: Отмена записи клиентом

- **Актёры:** Клиент
- **Предусловия:** У клиента есть активная бронь на предстоящий заезд.
- **Триггер:** Клиент решает отменить участие.
- **Приоритет:** Must
- **Источник:** [domain-description.md](../1-elicitation/domain-description.md) → «Процесс», «Бизнес-ограничения», «Глоссарий» → «Отмена»
- **Связанные FR:** FR-15, FR-16, FR-17

### Основной поток
1. Клиент открывает «Мои бронирования» и выбирает предстоящую бронь.
2. Нажимает «Отменить».
3. **Ранняя отмена (≥1 ч до старта):** бронь получает статус «отменена клиентом»; места и прокатный фонд освобождаются.
4. Система показывает подтверждение отмены.

### Альтернативные потоки
- **A1. Поздняя отмена (<1 ч до старта).** Система показывает предупреждение; отмена выполняется; отмена учитывается в статистике поздних отмен центра; денежных штрафов и блокировок нет.

### Исключения / ошибки
- **E1. Заезд уже начался или завершён.** Отмена недоступна.

### Постусловия
- Статус брони обновлён; при ранней отмене места снова доступны другим клиентам.

## UC-5: Отмена заезда центром (отображение для клиента)

- **Актёры:** Клиент; существующая инфраструктура (инициатор отмены — вне скоупа клиента)
- **Предусловия:** У клиента есть активная бронь; центр отменил заезд (организатор / погода).
- **Триггер:** Центр отменил слот в существующей инфраструктуре (R-008).
- **Приоритет:** Must
- **Источник:** [domain-description.md](../1-elicitation/domain-description.md) → «Бизнес-ограничения» (R-008), «Глоссарий» → «Отмена центром»
- **Связанные FR:** FR-18, FR-21

### Основной поток
1. Бэкенд переводит бронь клиента в статус «Отменён центром» с указанием причины.
2. Клиент получает push-уведомление.
3. Клиент видит бронь в «Моих бронированиях» со статусом и причиной.
4. Повторная запись на этот слот недоступна.

### Постусловия
- Бронь сохранена в истории клиента; слот недоступен для новой записи.

## UC-6: Оценка маршала после заезда

- **Актёры:** Клиент
- **Предусловия:** У клиента есть завершённая бронь; оценка ещё не была оставлена.
- **Триггер:** Клиент хочет оценить маршала после состоявшегося заезда.
- **Приоритет:** Must
- **Источник:** [domain-description.md](../1-elicitation/domain-description.md) → «Процесс», «Ключевые объекты домена» → «Оценка маршала»
- **Связанные FR:** FR-19, FR-5

### Основной поток
1. Клиент открывает завершённую бронь.
2. (Опц.) Оставляет оценку маршала.
3. Система сохраняет оценку **однократно**, без возможности редактирования.
4. Средний рейтинг маршала обновляется и отображается в карточках слотов.

### Альтернативные потоки
- **A1. Клиент пропускает оценку.** Оценка необязательна; бронь остаётся завершённой без оценки.

### Постусловия
- При оценке: сохранена разовая обратная связь; средний рейтинг маршала доступен в карточках слотов.

## UC-7: Push-напоминание перед заездом

- **Актёры:** Клиент
- **Предусловия:** У клиента есть предстоящая активная бронь.
- **Триггер:** Наступает время отправки напоминания (логика — на стороне существующей инфраструктуры).
- **Приоритет:** Should
- **Источник:** [domain-description.md](../1-elicitation/domain-description.md) → «Пользователи и роли», «Границы скоупа»
- **Связанные FR:** FR-20

### Основной поток
1. Система отправляет клиенту push-напоминание о предстоящем заезде.
2. Клиент видит уведомление на телефоне.

### Постусловия
- Клиент проинформирован о предстоящем заезде.
