# Промпты генерации `analysis/` — картинг-центр «Апекс»

> Мастер-документ реконструирует последовательность запросов к ИИ-аналитику/дизайнеру,
> которыми была собрана директория `analysis/` проекта «Апекс» (клиентское мобильное
> приложение для записи на заезды картинг-центра). Каждый раздел ниже соответствует ровно
> одной подпапке `analysis/` и представляет собой промпт, по которому эта подпапка была
> заполнена. Разделы идут в реальном порядке зависимостей: каждый следующий использует как
> источник материалы, созданные на предыдущих шагах.

**Роль во всех запросах:** Ты — ИИ бизнес- и системный аналитик уровня senior с опытом
проектирования мобильных B2C-продуктов и написания технической документации для разработки.

**Сквозной контекст проекта** (передаётся во все запросы ниже):
Уличный картинг-центр «Апекс» третий сезон ведёт запись на групповые заезды вручную
(Telegram + доска маркером), что приводит к двойным броням. Нужно клиентское мобильное
приложение для самостоятельной записи, читающее расписание/трассы/маршалов из уже
существующего бэкенда (read-only) и управляющее только бронями, оценками маршала и
частично профилем клиента. Срок — ~2 месяца, бюджет ограничен, MVP-подход.

## Запрос 1 — `analysis/1-elicitation/`

**Файлы:** `customer-questions.md`, `domain-description.md`

**Источник:** `analysis/0-customer-brief/brief-karting.md`.

**Промпт:**

Это этап выявления требований. Работай в два шага и оформи их как два отдельных файла.

**Шаг А — `customer-questions.md`.**
Прочитай бриф заказчика и найди все места, где он неполон или противоречив. Сформулируй
уточняющие вопросы аналитика, сгруппированные по тематическим блокам (бронирование,
экипировка, оценка маршала, уведомления, оплата, профиль клиента и ограничения картинга,
постоянные клиенты). Явно укажи в примечании, что вопросы про границы скоупа (роли,
формирование расписания, канонические данные, горизонт списка слотов, поведение при
отмене центром) уже закрыты в брифе через R-004/R-008/R-015/R-027/R-028 и повторно не
задаются. Затем для каждого вопроса зафиксируй раздел «Ответы заказчика», где в формате
«вопрос → варианты, предложенные аналитиком → выбор заказчика» дай финальный ответ и
кратко опиши, как он повлияет на будущие требования.

**Шаг Б — `domain-description.md`.**
На основе брифа и полученных ответов собери согласованное, уже непротиворечивое описание
домена: бизнес и процесс своими словами, список ключевых объектов домена (заезд/слот,
конфигурация трассы, маршал, карт, экипировка, бронь, оценка маршала, клиент), таблицу
«Пользователи и роли» с явной пометкой, что в текущий скоуп входит только роль «Клиент»,
раздел «Ограничения» (бизнес-ограничения, технические/платформенные, проектные), таблицу
«Границы скоупа» (что в скоупе клиентского приложения / что вне скоупа) и глоссарий терминов
домена. Все ограничения, пришедшие из R-004/R-008/R-015/R-027/R-028, должны быть здесь
явно поименованы и раскрыты подробно — это финальный источник истины по домену для всех
последующих этапов.

## Запрос 2 — `analysis/2-requirements/`

**Файлы:** `business-requirements.md`, `functional-requirements.md`,
`non-functional-requirements.md`, `use-cases.md`, `user-stories.md`

**Источник:** `analysis/1-elicitation/domain-description.md`.

**Промпт:**

На основе согласованного описания домена сформируй полный комплект требований второго
этапа, единообразно ссылающийся на `domain-description.md` как источник для каждой строки.

1. **`business-requirements.md`** — бизнес-требования верхнего уровня (BR-*) с приоритетом
   по MoSCoW, отдельный раздел метрик успеха (число двойных броней → 0; готовность к
   старту сезона) и раздел «Ограничения верхнего уровня», дублирующий ключевые бизнес-
   ограничения домена для быстрой сверки.
2. **`functional-requirements.md`** — функциональные требования (FR-*), сгруппированные по
   разделам домена (авторизация; просмотр и фильтрация слотов; запись на заезд; мои
   бронирования и отмены; оценка маршала; уведомления), с приоритетом MoSCoW и явной
   ссылкой на конкретный абзац `domain-description.md` в колонке «Источник». В конце —
   отдельный блок «Вне скоупа (Won't)».
3. **`non-functional-requirements.md`** — нефункциональные требования (NFR-*) отдельной
   таблицей по категориям (удобство использования, интеграция, надёжность/целостность
   данных, безопасность/аутентификация, разграничение доступа, ограничения проекта,
   архитектура/границы), тоже со ссылками на домен.
4. **`use-cases.md`** — сценарии использования (UC-*) в классическом формате (актёры,
   предусловия, триггер, приоритет, источник, связанные FR, основной поток, альтернативные
   потоки, исключения/ошибки, постусловия) для каждого крупного сценария: авторизация,
   просмотр/фильтрация слотов, запись (себя и гостей), отмена клиентом, отмена центром
   (только отображение на стороне клиента), оценка маршала, push-напоминание.
5. **`user-stories.md`** — user stories в формате «Как <роль>, я хочу <действие>, чтобы
   <ценность>» с критериями приёмки в отдельной колонке и ссылкой на соответствующий FR.

Все пять файлов должны быть терминологически согласованы друг с другом и с
`domain-description.md` — не вводи новые названия сущностей или статусов, которых там нет.

## Запрос 3 — `analysis/3-design-brief/`

**Файлы:** `design-brief.md`, `00-foundations.md`

**Источники:** `analysis/1-elicitation/domain-description.md`,
`analysis/2-requirements/*.md`.

**Промпт:**

Подготовь бриф для UI/UX-дизайнера в двух документах: сквозные правила и обзорный индекс.

**`00-foundations.md`** — сквозной документ дизайн-принципов, обязателен к прочтению перед
любым экранным документом:
- продукт и аудитория (mobile-first, использование у трассы на солнце);
- 5–6 именованных дизайн-принципов (P1…P6) с явной ссылкой на конкретные NFR/FR/домен;
- структурные токены без привязки к бренду (тач-зоны ≥44pt, контраст WCAG AA, типографическая
  иерархия, плотность/сетка, иконки/индикаторы) — визуальный язык (цвета, шрифты) оставь
  на усмотрение дизайнера;
- каркас экрана и навигация: базовая раскладка, таб-бар (Заезды / Мои брони), правила
  bottom sheet (высота, свайп, бэкдроп, исключение для критичных подтверждений),
  переиспользуемый блок «Место сбора и адрес центра»;
- единый сквозной паттерн состояний экрана (Loading / Content / Empty / Error);
- tone of voice и общая микрокопия — единая таблица формулировок (оплата, экипировка,
  правило отмены, статусы, ошибки), чтобы каждый экранный документ мог ссылаться на неё
  вместо дублирования текста;
- раздел доступности;
- сквозные функции (push-разрешение только после первой брони; безопасность данных в UI);
- глоссарий UI-терминов;
- карта документов — таблица со ссылками на все экраны, которые будут описаны позже
  (`../screens/SCR-00X...md`, `BS-00X...md`) — на этом этапе достаточно зарезервировать ID
  и имена файлов, сами документы появятся на следующем шаге.

**`design-brief.md`** — короткий индекс поверх foundations: цель и контекст продукта, что
входит в MVP (списком, с ссылками на FR), таблица «Роли и их задачи» (только Клиент — в
скоупе; Маршал/Владелец — явно вне скоупа), таблица экранов (ID, тип, зона авторизации,
приоритет, ссылка на будущий файл в `../screens/`), пользовательские сценарии (user flows)
верхнего уровня со ссылками на UC-* из `2-requirements/use-cases.md`, краткий раздел
«Тон, бренд, ограничения» со ссылкой на `00-foundations.md`.

## Запрос 4 — `analysis/4-design/`

**Файлы:** `api-sequence.md`, `data-model.md`

**Источники:** `analysis/2-requirements/*.md`, `analysis/1-elicitation/domain-description.md`,
`analysis/3-design-brief/00-foundations.md`.

**Промпт:**

Спроектируй информационную модель и ключевую последовательность API-взаимодействия ещё до
того, как будут расписаны конкретные экраны и логики — эти два документа станут источником
истины для контракта API, экранных документов и переиспользуемых логик.

**`data-model.md`**:
- таблицы атрибутов для каждой сущности домена (Client, TrackConfiguration, Marshal, Slot,
  Booking, Rating) с колонками «Атрибут / Тип / Режим (Read-only, Write-once, Write/Read) /
  Описание», явно помечая, какие сущности read-only для клиентского приложения, а какие
  им управляются;
- ERD-диаграмма (mermaid `erDiagram`) со всеми связями;
- диаграмма жизненного цикла брони (mermaid `flowchart`) со всеми переходами статуса
  (active → cancelled / late_cancel / center_cancelled) и таблица эффекта каждого перехода
  на инвентарь слота (карты/экипировка);
- раздел «Ключевые инварианты целостности данных», пронумерованный списком: расчёт
  свободных мест, жёсткий потолок конфигурации трассы, расчёт остатка экипировки, условия
  создания оценки маршала, разница между early/late отменой, атомарность записи/отмены.

**`api-sequence.md`**:
- раздел «Сквозные технические правила» (авторизация через Bearer/401, идемпотентность
  через заголовок Idempotency-Key, атомарность транзакции на сервере, офлайн-расчёт
  стоимости) — вывести напрямую из NFR и R-004 домена;
- одна детальная sequence-диаграмма (mermaid `sequenceDiagram`) сценария создания брони
  (`POST /bookings`), покрывающая happy path и все ветки бизнес-ошибок: конфликт мест,
  конфликт экипировки, слот отменён администрацией, слот уже начался, ошибка авторизации,
  сетевой сбой/5xx;
- таблица-спецификация «HTTP статус / код ошибки / описание сценария / реакция UI»,
  дословно согласованная с диаграммой — она станет черновиком для `analysis/api/openapi.yaml`
  и для разделов «Используемые запросы» экранных документов.

## Запрос 5 — `analysis/api/`

**Файлы:** `openapi.yaml`

**Источники:** `analysis/4-design/data-model.md`, `analysis/4-design/api-sequence.md`,
`analysis/2-requirements/functional-requirements.md`,
`analysis/1-elicitation/domain-description.md`.

**Промпт:**

*(этот запрос воспроизводит переданный тебе `api-prompt.md` практически дословно — он и есть
промпт данного шага, с единственной поправкой: на момент выполнения `analysis/screens/` ещё
не существовало, поэтому источниками служат `4-design/` и `2-requirements/`, а не готовые
экраны)*

Собери единый файл `analysis/api/openapi.yaml` версии OpenAPI 3.0.3, покрывающий полный
контракт клиентского приложения:
- security scheme `bearerAuth` (JWT), применяемый ко всем путям кроме `/auth/*`;
- теги: Auth, Profile, Slots, Bookings, Instructors, Ratings;
- пути авторизации (`request-code`, `verify-code`, `refresh`, `logout`, `push-tokens`
  POST/DELETE), профиля (`GET/PATCH/DELETE /profile`, смена телефона), слотов
  (`GET /slots` с пагинацией и фильтрами `date_from/date_to/track_config/instructor_id/
  only_available`, `GET /slots/{slotId}`), броней (`POST /bookings` с обязательным
  заголовком `Idempotency-Key`, `GET /bookings`, `GET /bookings/{bookingId}`,
  `POST /bookings/{bookingId}/cancel`), маршалов (`GET /instructors`), оценок
  (`POST /ratings`, `GET /ratings?booking_id=`);
- переиспользуемые схемы: `Error` (с закрытым перечислением машинных кодов ошибок),
  `PaginationMeta`, `Client`, `TrackConfiguration`, `Marshal`, `Slot`/`SlotSummary`,
  `Booking`/`BookingSummary`, `RatingSummary`/`Rating`, все Request/Response DTO;
  переиспользуемые `responses` (BadRequest, Unauthorized, Forbidden, NotFound, Conflict,
  Gone, UnprocessableEntity, TooManyRequests, InternalError) с примерами `example:`,
  дословно согласованными с таблицей ошибок из `api-sequence.md`;
- в описании каждого пути и в комментарии `info.description` явно зафиксировать: скоуп —
  только клиентское приложение; слоты/трассы/маршалы приходят из существующего бэкенда и
  доступны только для чтения; сервер — источник истины по времени, доступности и цене;
  двойные брони и овербукинг исключены атомарно на сервере.

Не изобретай эндпоинтов сверх того, что подтверждается `data-model.md`/`api-sequence.md`/
функциональными требованиями. Формат — как в этом же промпте, который ты сейчас читаешь.

## Запрос 6 — `analysis/screens/`

**Файлы:** `SCR-001-registration.md`, `SCR-002-slot-list.md`, `SCR-003-slot-card.md`,
`SCR-004-booking.md`, `SCR-005-my-bookings.md`, `SCR-006-booking-details.md`,

**Источники:** `analysis/3-design-brief/00-foundations.md`,
`analysis/3-design-brief/design-brief.md`, `analysis/4-design/data-model.md`,
`analysis/4-design/api-sequence.md`, `analysis/api/openapi.yaml`,

**Промпт:**

Для каждого ID экрана/шторки из таблицы «Экраны» в `design-brief.md` подготовь отдельный
файл по единому шаблону с разделами: Обзор (+ User Story, Бизнес-ценность), Навигация
(входящая/исходящая с параметрами перехода), Входные данные, Применяемые логики (пока
ссылками-заглушками на будущие `../09-logic/LOGIC-00X-*.md` — они будут вынесены отдельным
шагом), Инициализация (диаграмма загрузки mermaid + таблица запросов при открытии),
Используемые запросы (со ссылкой на конкретный `operationId` из `openapi.yaml`), Макет
экрана (ASCII-каркас + таблица компонентов), Элементы экрана (таблица «Элемент / Описание /
Источник данных / Валидация / Действие» на каждый смысловой блок), Состояния экрана
(таблица состояний + диаграмма переходов mermaid `stateDiagram-v2`), Действия пользователя,
Связанные требования (REQ-FUNC-*/REQ-INT-*/REQ-UI-*/REQ-DATA-* с приоритетом), Критерии
приёмки (позитивные/негативные/граничные, с приоритетом P0–P2).

Сгенерируй в следующем порядке, поддерживая сквозной flow записи и управления бронью:

1. `SCR-001-registration.md` — вход по телефону + SMS-код, без пароля (два шага на одном
   экране), единственная неавторизованная зона (НЗ).
2. `SCR-002-slot-list.md` — стартовая вкладка «Заезды», список слотов на 7 дней по
   умолчанию, фильтры, пагинация, empty/error state.
3. `BS-001-filters.md` — шторка фильтров (дата/трасса/наличие мест/маршал), не закрывается
   по тапу-в-пустоту исключений из этого правила нет.
4. `SCR-003-slot-card.md` — полная карточка слота перед записью, расчёт доступности
   CTA «Записаться» через будущую LOGIC-002.
5. `SCR-004-booking.md` — оформление записи: степпер мест, переключатели «своя/прокат» на
   каждое место, справочная цена до отправки, финальная — из ответа сервера.
6. `BS-002-booking-success.md` — подтверждение записи + точка запроса push-разрешения
   (только на первой успешной записи за всё время использования аккаунта).
7. `SCR-005-my-bookings.md` — список броней, разделение на предстоящие/прошедшие.
8. `SCR-006-booking-details.md` — детали брони, вариативный CTA по статусу (отменить /
   оценить маршала / блок «Ваша оценка» / нет CTA).
9. `BS-003-cancel-confirm.md` — критичное подтверждение отмены (не закрывается тапом
   вне области), с локальным предупреждением о правиле «1 час», без штрафов.
10. `BS-004-marshal-rating.md` — необязательная однократная оценка маршала (1–5 звёзд +
    комментарий до 500 символов), без возможности редактирования после отправки.

Каждый экран должен явно ссылаться на текст правил из `00-foundations.md` (например,
формулировку правила отмены) вместо повторного изложения — избегай дублирования микрокопии
между файлами.

## Запрос 7 — `analysis/09-logic/`

**Файлы:** `_INDEX.md`, `LOGIC-001-auth-otp.md`, `LOGIC-002-availability-calc.md`,
`LOGIC-003-create-booking.md`, `LOGIC-004-cancel-booking.md`,
`LOGIC-005-marshal-rating.md`, `LOGIC-006-push-notifications.md`

**Источники:** `analysis/screens/*.md` (уже готовые к этому моменту), `analysis/api/openapi.yaml`,
`analysis/4-design/api-sequence.md`, `analysis/4-design/data-model.md`.

**Промпт:**

При написании экранов в `analysis/screens/` одна и та же бизнес-логика (авторизация,
расчёт доступности, создание/отмена брони, оценка маршала, push) описывалась почти
дословно в нескольких файлах одновременно — вынеси её в отдельные переиспользуемые
документы по принципу DRY, чтобы экраны ссылались на неё, а не дублировали.

Для каждой логики подготовь файл по шаблону: Обзор (+ User Story, Бизнес-ценность), Точки
применения (таблица «Экран/Компонент / Элемент-триггер / Условие», перечисляющая все
экраны из предыдущего шага, где эта логика реально использовалась), Флоу (mermaid
`flowchart`, покрывающий все ветки успеха/ошибок из `api-sequence.md`/`openapi.yaml`),
Описание логики (пронумерованные шаги), API запросы (с точной таблицей параметров и
обработкой каждого кода ответа — синхронизировано с `openapi.yaml` дословно), Связанные
требования, Критерии приёмки.

Выдели ровно шесть логик, обосновав каждую конкретным дублированием, найденным в экранах:
- **LOGIC-001** (OTP-авторизация и сессия) — используется на SCR-001 и как сквозной
  interceptor на всех авторизованных экранах;
- **LOGIC-002** (расчёт доступности мест и проката) — дублировалось на SCR-003 и SCR-004;
- **LOGIC-003** (создание брони) — идемпотентность и обработка конфликтов с SCR-004;
- **LOGIC-004** (отмена брони) — локальное предупреждение о часе + отправка с SCR-006/BS-003;
- **LOGIC-005** (оценка маршала) — вычисление «завершённости» заезда на клиенте (в API нет
  явного статуса `completed`) с SCR-005/SCR-006/BS-004;
- **LOGIC-006** (push-уведомления) — момент запроса разрешения, регистрация токена,
  разбор входящих push с BS-002 и глубоких ссылок из push на SCR-006.

Присвой файлам числовой префикс `09-`, чтобы при сортировке по имени папка отображалась
после `screens/`, несмотря на то что логики применяются везде, где ссылаются экраны —
это сознательное решение, а не ошибка нумерации.
