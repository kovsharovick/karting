# Архитектурный план клиентского приложения «Апекс»

Версия: 1.0  
Дата: 2026-07-04  
Статус: Черновик

## 1. Введение

### 1.1 Цель документа
Описать архитектуру клиентского мобильного приложения для картинг-центра «Апекс» и сопутствующей серверной инфраструктуры (БД, API). Основной фокус – на клиентском приложении и его интеграции с существующим бэкендом.

### 1.2 Контекст
«Апекс» – уличный картинг-центр, работающий третий сезон. Сейчас запись ведётся вручную (Telegram + доска), что приводит к двойным броням и путанице. Задача – создать мобильное приложение для самостоятельной записи клиентов.

**Скоуп текущей работы**:
- Клиентское мобильное приложение (iOS/Android).
- API для него (контракт + логика на стороне клиента).
- База данных PostgreSQL.

**Вне скоупа** (существующая инфраструктура):
- Формирование расписания (слоты, трассы, маршалы).
- Интерфейсы маршала и администратора.
- Управление парком картов, онлайн-оплата, лояльность, погода.

## 2. Архитектурные принципы

| Принцип | Реализация |
|---------|------------|
| Mobile-first | Крупные тач-зоны, высокий контраст, минимум шагов. |
| Разделение ответственности | Бэкенд – источник истины по слотам; клиент – только брони/оценки. |
| Атомарность на сервере | Транзакции с SELECT FOR UPDATE при бронировании. |
| Идемпотентность | Idempotency-Key в POST /bookings. |
| Минимальный порог входа | Вход по телефону + OTP (SMS), без пароля. |
| Безопасность данных | JWT, refresh-токены в защищённом хранилище, только свои данные. |
| Прозрачность правил | Микрокопия о 1 часе, штрафов нет. |

## 3. Обзор системы

### 3.1 Компоненты
- **Клиент**: UI, бизнес-логика (LOGIC-001..006), локальное хранилище, push-сервис (APNs/FCM).
- **Бэкенд**: API Gateway, сервисы авторизации, бронирования, оценок, слотов (read-only), интеграция с админкой.
- **БД**: PostgreSQL с триггерами для атомарного резервирования и освобождения инвентаря.

### 3.2 Основные сценарии
1. **Авторизация**: запрос кода → подтверждение → JWT.
2. **Просмотр слотов**: GET /slots (7 дней по умолчанию) с фильтрацией.
3. **Запись**: POST /bookings с Idempotency-Key → атомарное списание инвентаря.
4. **Отмена**: POST /bookings/{id}/cancel – ранняя (возврат) или поздняя (без возврата).
5. **Оценка**: POST /ratings (только после завершения заезда).
6. **Push**: напоминание перед заездом и уведомление об отмене центром.

## 4. Модель данных

### 4.1 Сущности (ключевые)
- **Client**: id, phone (UK), name, created_at, deleted_at.
- **TrackConfiguration**: id, name, type (novice/experienced), max_group_size.
- **Marshal**: id, name, rating_avg, rating_count.
- **Slot**: id, track_config_id, marshal_id, start_at, duration_minutes, total_karts, free_karts, total_rental_gear, free_rental_gear, price_kart, price_gear_rental, requirements_text, meeting_point, address, status (scheduled/cancelled), cancellation_reason.
- **Booking**: id, slot_id, client_id, seats_count, rental_gear_count, status (active/cancelled/late_cancel/center_cancelled), price_total, cancellation_reason, idempotency_key, created_at, cancelled_at.
- **Rating**: id, client_id, marshal_id, booking_id (UK), value, comment, created_at.
- **PushToken**: id, client_id, token, platform.
- **OTPCode**: id, phone, purpose, code_hash, attempts, expires_at, resend_after, consumed_at.
- **RefreshToken**: id, client_id, token_hash, expires_at, revoked_at, replaced_by.
- **IdempotencyKey**: (idempotency_key, client_id, endpoint) PK, request_fingerprint, response_status, response_body, created_at.
- **PushNotificationLog**: id, client_id, booking_id, type, sent_at.

### 4.2 Ключевые инварианты (триггеры)
- `free_karts` >= 0, `total_karts` <= потолок трассы (8 для novice, 14 для experienced).
- Резервирование инвентаря при создании брони (SELECT FOR UPDATE).
- Освобождение только при ранней отмене (status = cancelled), при late_cancel – не освобождается.
- Оценка только для активной брони после `start_at + duration_minutes`.
- Одна оценка на бронь (UNIQUE).
- Одна активная бронь клиента на слот (UNIQUE INDEX WHERE status='active').
- Идемпотентность через таблицу idempotency_keys.

## 5. API (основные эндпоинты)

| Метод | Путь | Описание |
|-------|------|----------|
| POST | /auth/request-code | Запросить OTP-код |
| POST | /auth/verify-code | Подтвердить код, получить JWT |
| POST | /auth/refresh | Обновить access-токен |
| POST | /auth/logout | Завершить сессию |
| GET | /slots | Список слотов (фильтры, пагинация) |
| GET | /slots/{slotId} | Детали слота |
| POST | /bookings | Создать бронь (Idempotency-Key) |
| GET | /bookings | Список броней клиента |
| GET | /bookings/{bookingId} | Детали брони |
| POST | /bookings/{bookingId}/cancel | Отменить бронь |
| POST | /ratings | Оценить маршала |
| GET | /ratings?booking_id= | Получить оценку по брони |
| POST | /auth/push-tokens | Зарегистрировать push-токен |
| DELETE | /auth/push-tokens | Отвязать push-токен |
| GET | /profile | Получить профиль |
| PATCH | /profile | Обновить имя |
| DELETE | /profile | Удалить аккаунт |

**Важно**: все запросы (кроме /auth) требуют `Authorization: Bearer <access_token>`. Ошибки возвращаются в формате `{code, message, details}`.

## 7. Инфраструктура

### 7.1 База данных
- PostgreSQL 16 (образ postgres:16-alpine).
- Развёртывание: Docker Compose (database/docker-compose.yml).
- Миграции: сейчас через docker-entrypoint-initdb.d (V1__init_schema.sql), в будущем – Flyway.
- Переменные окружения: .env (пароль, порты).
- Том: karting_pgdata для сохранения данных.
- Adminer – опционально через профиль tools.

### 7.2 Бэкенд (предполагаемый)
- Java/Kotlin + Spring Boot.
- REST по OpenAPI.
- Работа с БД через JPA/Exposed/jOOQ.
- Транзакции, маппинг ошибок триггеров на HTTP-статусы.
- Flyway для миграций (после перехода).

### 7.3 Клиентское приложение
- iOS (Swift) / Android (Kotlin) – нативное или кросс-платформа.
- Сетевой слой: REST-клиент с Idempotency-Key, интерцептором для refresh, обработкой ошибок.
- Хранилище: Keychain/Keystore для токенов, локальный кэш (Room/CoreData).
- Push: APNs/FCM, запрос разрешения после первой записи.
- Навигация: таб-бар (Заезды / Мои брони), стеки, bottom sheets.
- UI: по дизайн-системе (00-foundations.md).

## 8. Безопасность

- Аутентификация: телефон + OTP (без пароля).
- Сессии: JWT (короткий) + refresh-токен (хэш в БД, ротация).
- Хранение токенов: защищённое хранилище ОС.
- Защита от перебора OTP: attempts, rate-limit.
- Защита от двойных броней: идемпотентность + атомарные транзакции.
- Конфиденциальность: клиент видит только свои данные, телефон маскируется.
- Push: токены привязаны к клиенту, удаляются при выходе.
