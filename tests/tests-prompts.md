# Промпты генерации `tests/` — тест-кейсы и баг-репорты для веб-приложения картинг-центра «Апекс»

> Мастер-документ реконструирует последовательность запросов к ИИ QA-аналитику,  
> по которым была собрана директория `tests/` проекта «Апекс».  
> Сначала были подготовлены ручные тест-кейсы для проверки критических сценариев,  
> затем по результатам ручного прогона были оформлены баг-репорты на найденные дефекты.

**Роль во всех запросах:**  
Ты — senior QA engineer / QA analyst с опытом тестирования Spring Boot + JSP веб-приложений, REST API, пользовательских сценариев бронирования, OTP-авторизации и проверки бизнес-логики на уровне UI + backend.

**Сквозной контекст проекта:**  
Разрабатывается веб-приложение картинг-центра «Апекс» для самостоятельной записи клиентов на групповые заезды.  
Пользователь входит по номеру телефона и OTP-коду, просматривает список доступных слотов, фильтрует заезды, открывает карточку слота, создаёт бронь на себя и гостей, видит свои брони, отменяет бронь, а после завершения заезда может оценить маршала.

Приложение реализовано на **Java 17 + Spring Boot + Spring MVC + JSP + Spring Data JPA**.  
В dev-профиле используется H2, тестовые данные создаются через initializer/миграции.  
Ключевые сущности: клиент, OTP-код, слот, конфигурация трассы, маршал, бронь, оценка маршала.

Критичные бизнес-правила:

- пользователь авторизуется без пароля через OTP;
- один пользователь не должен создавать дублирующие активные брони на один и тот же слот;
- количество свободных картов и прокатной экипировки должно уменьшаться при создании брони;
- количество свободных ресурсов должно корректно возвращаться при допустимой отмене;
- фильтрация слотов должна работать по датам, типу трассы, маршалу и доступности;
- страница деталей брони не должна падать при отсутствующих или неполных данных;
- UI должен всегда давать авторизованному пользователю доступ к разделу «Мои брони».
